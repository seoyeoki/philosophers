# 📚 Philosophers 프로젝트 핵심 개념 정리

Philosophers 프로젝트의 주요 스케줄링 이론, 다중 스레드 동기화 기법, 데이터 레이스 해결 방안 및 테스트 검증 절차를 정리한 문서입니다.

---

## 📌 1. Mandatory와 Bonus의 차이 (메모리와 동기화)

| 구분 | Mandatory | Bonus |
| :--- | :--- | :--- |
| **기본 단위** | **스레드 (Thread)** | **프로세스 (Process)** |
| **메모리 구조** | 하나의 주소 공간을 **모든 스레드가 공유** | `fork()`로 생성되어 **독립된 가상 메모리**를 가짐 |
| **동기화 도구** | **Mutex** (`pthread_mutex_t`) | **Named Semaphore** (`sem_t`) |
| **자원 관리** | 스레드 메모리 누수(`pthread_join`) 및 락 경합 방지 필요 | 세마포어 전역 해제 및 프로세스 좀비화 방지(`waitpid`) 필요 |

## 📌 2. Mutex를 통한 데이터 레이스 방지 (읽기/쓰기 대칭성)

데이터 레이스(Data Race)를 원천 차단하기 위해서는 **공유 데이터를 변경할 때(Write)뿐만 아니라 조회할 때(Read)도 동일한 Mutex로 보호**해야 합니다.

### ❌ Mutex가 없는 위험한 버전 (Data Race 발생)
스레드 A가 값을 쓰는 동안 스레드 B가 동시에 읽으려고 시도할 때, 데이터가 불완전한 상태로 읽히는 데이터 레이스가 발생합니다.
```c
/* Thread A (철학자 식사 시작 시점 기록) */
philo->last_meal_time = get_time();

/* Thread B (모니터링 스레드 사망 여부 확인) */
time_since_meal = get_time() - philo->last_meal_time;
```

###  Mutex로 안전하게 감싼 버전 (대칭 락 적용)
동일한 `meal_mutex`를 양쪽에 적용하여 한쪽이 기록하는 동안 다른 쪽이 접근하지 못하도록 제어합니다.
```c
/* Thread A (철학자 식사 시작 시점 기록) */
pthread_mutex_lock(&philo->meal_mutex);
philo->last_meal_time = get_time();
pthread_mutex_unlock(&philo->meal_mutex);

/* Thread B (모니터링 스레드 사망 여부 확인) */
pthread_mutex_lock(&philo->meal_mutex);
time_since_meal = get_time() - philo->last_meal_time;
pthread_mutex_unlock(&philo->meal_mutex);
```

> [!NOTE]  
> **사망(dead)과 출력(print)의 동시 관리**: 철학자의 사망 플래그(`over` 또는 `dead`)를 갱신하는 시점과 "died" 메시지를 화면에 출력하는 과정은 원자적(Atomic)으로 묶여야 합니다. 이를 분리하면 철학자가 사망한 뒤에도 다른 철학자의 행동 로그가 추가로 출력되는 규정 위반이 발생하므로, 반드시 하나의 `print_mutex` 안에서 안전하게 묶어 관리해야 합니다.

## 📌 3. 바쁜 대기(Busy Wait)와 스레드 양보

* **바쁜 대기**: 스레드가 CPU 자원을 100% 사용하면서 빈 루프(`while (1)`)를 계속 도는 현상입니다.
* **현상**: 한 철학자 스레드가 바쁘게 루프를 돌며 자원을 선점하려 하면, 컨텍스트 스위칭이 늦어져 모니터 스레드가 사망 여부를 감시할 타이밍을 놓치게 됩니다. (다만 CPU 코어 수가 많을 때는 스레드 분배가 분산되어 정상 작동하는 것처럼 보일 수 있습니다.)
* **해결**: 루프 내에 미세한 `usleep` 지연(예: `usleep(1000)`)을 추가하여 CPU 소유권을 다른 스레드에 넘겨주는 **스레드 양보(Yield)** 방식을 적용해야 합니다.

## 📌 4. 모니터와 철학자 스레드 아키텍처

시뮬레이션에서 생성되는 **모든 개별 철학자의 행동 흐름과 감시용 모니터 루틴은 각각 독립적인 스레드**로 구동됩니다.
* **철학자 스레드**: 루프를 돌며 식사 ➔ 수면 ➔ 생각 행동을 반복적으로 수행합니다.
* **모니터 스레드**: 별도의 모니터링 루틴에서 철학자들의 `last_meal_time`을 순회 감시하며 한 명이라도 사망 시간을 넘었는지 실시간으로 추적합니다.

## 📌 5. 식사, 수면, 생각 시간의 조화와 스케줄링 공식

### A. 식사 스케줄링 설계와 위상차 공식
* **사망 계수**: $\frac{\text{전체 조 수}}{\text{동시에 식사 가능한 조 수}}$
  * 예를 들어, 5개 그룹으로 나눌 경우 한 번에 최대 2개 그룹이 동시에 식사를 진행할 수 있으므로 사망 계수는 2.5가 됩니다.
* **지연 템포**: $\text{조 번호} \times \frac{\text{time\_to\_eat}}{2}$
  * 기동 딜레이 간격은 식사 리듬 주기를 그룹 수에 맞게 분배한 값입니다.
* **왜 5조인가?**
  * 철학자의 인원수에 따라 5조, 7조, 11조, 199조 등으로 분할하는 방식을 고려할 수 있으나, 가장 최소 단위인 3조의 경우 인접 포크 경쟁 구조상 5조 스케줄링에서 3명만 구동되는 형태와 구조적으로 완벽히 동일합니다. 따라서 5조 스케줄링 구조를 적용하는 것이 대형 식탁($N=199$)까지 커버하면서도 가장 간결하고 최적화된 설계입니다.
* **임계 경계선 법칙**:
  * 식사 바통 터치의 구조상, $2 \times \text{time\_to\_eat} < \text{time\_to\_die} < 3 \times \text{time\_to\_eat}$ 사이의 좁은 수명 범위에서는 위상차 공식의 타이머 오차에 따라 생사 여부가 결정됩니다.

### B. Sleep 시간이 짧은 경우 (생각 완충 장치 작동)
* **짝수 조건**: $\text{time\_to\_die} > 2 \times \text{time\_to\_eat}$
* **홀수 조건**: $\text{time\_to\_die} > 2.5 \times \text{time\_to\_eat}$
  * 홀수 명의 상황에서는 3개의 시간대가 겹치므로 최소 2.5배의 마진이 필요합니다. 만약 임계선 부근($502\text{ms}$ 등)일 때는 OS의 CPU 스케줄러가 미세하게 튀는 노이즈(Jitter)로 인해 즉사할 수 있으므로, 미세 조율된 기동 지연(`init_delay`)과 생각 지연(`think_time`)을 통해 위상 템포를 끝까지 유지시켜야 합니다.

### C. Sleep 시간이 긴 경우 (의도적인 생각 지연 배제)
* **생존 조건**: $\text{time\_to\_die} > \text{time\_to\_sleep} + \text{time\_to\_eat}$
* 수면 시간 자체가 길어 포크를 내려놓는 시간이 널널한 경우, 별도의 생각 대기 시간(`think_time`)을 유도하지 않아도 충돌이 없어 100% 생존합니다.

## 📌 6. 누수 및 데이터 레이스 테스트 검증

리눅스 환경에서 메모리 및 스레드 경쟁 상태를 검증하기 위한 도구 사용법입니다.

1. **데이터 레이스 검사 (Helgrind)**:
   ```bash
   valgrind --tool=helgrind ./philo [인자값]
   ```
2. **OS 스케줄러 노이즈/시뮬레이션 공평성 제어 테스트**:
   Helgrind 검사 중 OS 스케줄러가 특정 스레드만 편애하여 편향되지 않도록 모든 스레드에 CPU 타임을 공평하게 할당하여 강도 높게 검증하는 옵션입니다.
   ```bash
   valgrind --tool=helgrind --fair-scheduled=yes ./philo [인자값]
   ```

## 📌 7. 평가 시 필수 테스트 커맨드 치트시트

평가(Defense) 진행 시 복사해서 바로 터미널에 붙여넣을 수 있는 명령어 리스트입니다.

### A. 표준 생존 및 즉사 테스트 케이스
* **1명 케이스 (즉사 확인)**: 800ms 부근에서 1번 사망 검증
  ```bash
  ./philo 1 800 200 200
  ```
* **짝수 명 생존 케이스**: 영원히 생존해야 함
  ```bash
  ./philo 4 410 200 200
  ```
* **짝수 명 경쟁 즉사 케이스**: 포크 선점으로 인해 310ms 부근에서 즉사
  ```bash
  ./philo 4 310 200 100
  ```
* **홀수 명 대형 식탁 케이스 (완료 조건)**: 199명 식탁에서 5회씩 안전하게 먹고 성공 종료
  ```bash
  ./philo 199 599 200 200 5
  ```

### B. macOS 전용 검사 커맨드 (누수 & 데이터 레이스)
* **컴파일러 기반 데이터 레이스 추적 (Thread Sanitizer)**:
  ```bash
  gcc -Wall -Wextra -Werror -fsanitize=thread -pthread *.c -o philo && ./philo 5 800 200 200 7
  ```
* **컴파일러 기반 메모리 누수 추적 (Address Sanitizer)**:
  ```bash
  gcc -Wall -Wextra -Werror -fsanitize=address -pthread *.c -o philo && ./philo 5 800 200 200 7
  ```
* **macOS 내장 leaks 명령어 활용 (실행 종료 시 즉시 검사)**:
  ```bash
  MallocStackLogging=1 leaks --atExit -- ./philo 5 800 200 200 7
  ```

### C. 리눅스(Linux) 환경 검사 커맨드
* **Valgrind를 이용한 메모리 누수 검사**:
  ```bash
  valgrind --tool=memcheck --leak-check=full --show-leak-kinds=all ./philo 5 800 200 200 7
  ```
* **Helgrind를 이용한 동기화/데이터 레이스 검사**:
  ```bash
  valgrind --tool=helgrind ./philo 5 800 200 200 7
  ```
* **OS 스케줄러 편애 방지 강도 테스트 (Helgrind)**:
  ```bash
  valgrind --tool=helgrind --fair-scheduled=yes ./philo 5 800 200 200 7
  ```
